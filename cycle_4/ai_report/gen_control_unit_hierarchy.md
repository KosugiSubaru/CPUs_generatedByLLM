「control_unit」はCPUの「司令塔」であり、命令の種類（opcode）を判別してデータパスを切り替える役割を持ちます。

ISAと回路の対応関係を視覚化するため、**「16種類の命令をそれぞれ個別に検出するデコーダー部」**と、**「検出結果から制御信号を生成するエンコーダー部」**に分割し、パタン構造化を適用します。

### 1. 階層構造の概要

| 階層レベル | ファイル名 | モジュール名 | 役割 |
| :--- | :--- | :--- | :--- |
| Level 0 (最小単位) | `control_unit_opcode_match.v` | `control_unit_opcode_match` | 入力されたopcodeが、特定の期待値（パラメータ）と一致するかを判定する比較器。 |
| Level 1 (最上位) | `control_unit.v` | `control_unit` | `generate`文で16個の`match`モジュールを並列展開し、その結果（どの命令か）を論理和でまとめて各制御信号を出力する。 |

---

### 2. 各モジュールの役割と説明

#### Level 0: control_unit_opcode_match.v
このモジュールは、4ビットのopcodeと、パラメータとして与えられた「ターゲットopcode」を比較します。
論理合成後の回路図では、特定の命令（例：`add`）のときにのみ「1」を出力する独立したブロックとして視覚化されます。

#### Level 1: control_unit.v
Level 0のモジュールを `generate` 文を用いて16個（opcode "0000" ～ "1111"）インスタンス化します。
- **デコード視覚化**: 16個のブロックが並び、現在実行中の命令に対応するブロックの出力だけが「1」になる様子を確認できます。
- **信号生成**: 各命令の判定結果を束ねて、以下のような制御信号を生成します。
    - `o_reg_write`: レジスタ書き込みを行う命令群（add, sub, load, jal等）の論理和。
    - `o_mem_write`: メモリ書き込みを行う命令（store）の信号。
    - `o_alu_src`: ALUの入力をレジスタにするか即値にするかの選択信号。

---

### 3. パタン構造化による教育的メリット

1.  **デコードプロセスの可視化**:
    「4ビットの数値」が「16本の独立した命令信号」へ変換される過程が回路図上で16個の並列ブロックとして表現されるため、デコーダーの仕組みが直感的に理解できます。
2.  **命令と制御信号の依存関係**:
    例えば「`o_reg_write` 信号は、どの命令ブロックとつながっているか？」を追うことで、どの命令がレジスタを書き換えるのかを回路図から読み取ることができます。
3.  **疎結合な設計**:
    新しい命令を追加したい場合、新しい `match` モジュールを追加して論理和に加えるだけで済むため、設計の定石（拡張性）を学ぶことができます。

### 4. 制御信号の対応（実装イメージ）
論理合成時には、以下のような対応関係が視覚化されるように構成します。
- **ALU演算系 (0000-1000)** → `o_reg_write` = 1, `o_alu_op` = opcode
- **Load (1010)** → `o_reg_write` = 1, `o_mem_to_reg` = 1
- **Store (1011)** → `o_mem_write` = 1
- **Branch (1100-1101)** → `o_branch_en` = 1
- **Jump (1110-1111)** → `o_jump_en` = 1, `o_reg_write` = 1 (戻りアドレス保存)
