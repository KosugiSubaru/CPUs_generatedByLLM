「imm_gen（即値生成ユニット）」の階層化とパターン構造化を提案します。

このユニットの役割は、命令コード内の異なる位置にあるビットフィールド（4bit, 8bit, 12bit）を抽出し、一律に16ビットへ符号拡張することです。ISAの各フォーマット（I型、S型、U型、B型）と回路の対応を視覚化するため、**「ビット単位の拡張論理」**から**「フォーマット別の拡張器」**へと階層化します。

---

### 1. 階層構造の定義

「符号拡張とは、特定のビットより上位を符号ビットで埋めることである」という概念を回路図で表現するため、以下の3レベルで構成します。

| 階層レベル | ファイル名 | 役割 |
| :--- | :--- | :--- |
| **Level 2 (Leaf)** | `imm_gen_bit_ext.v` | 1ビット分の拡張論理。そのビットが「データ範囲内」なら命令のビットを、「拡張範囲」なら符号ビットを出力する最小単位。 |
| **Level 1 (Structure)** | `imm_gen_sext4_i.v` | `imm_gen_bit_ext` を **generate文で16個並列化**。I型（addi等）の [7:4] を16bit化。 |
| | `imm_gen_sext4_s.v` | 同上。S型（store）の [15:12] を16bit化。 |
| | `imm_gen_sext8_u.v` | 同上。U型（loadi等）の [11:4] を16bit化。 |
| | `imm_gen_sext12_b.v` | 同上。B型（branch）の [15:4] を16bit化。 |
| **Level 0 (Top)** | `imm_gen.v` | 各フォーマットの拡張器を統合し、命令のオペコード（または制御信号）に応じて適切な即値を選択・出力する。 |

---

### 2. 各ファイルの役割と階層レベルの詳細

#### (1) imm_gen_bit_ext.v (Level 2)
符号拡張の「1ビット分」の振る舞いを定義します。
- **入力**: 命令からの1ビット（`i_inst_bit`）、符号ビット（`i_sign_bit`）、現在地がデータ範囲外かを示すフラグ（`i_is_ext_zone`）
- **出力**: 16ビット即値の構成要素となる1ビット
- **教育的意図**: 回路図上で、上位ビットが符号ビットに差し替えられる様子を視覚化します。

#### (2) imm_gen_sext[N]_[Type].v (Level 1)
各フォーマットに対応した16ビット生成器です。
- **手法**: `generate` 文を用いて、16個の `imm_gen_bit_ext` をインスタンス化します。
- **構造例 (I型)**: 
    - bit 0~3: `i_is_ext_zone` を 0 に固定（データビットをそのまま通す）
    - bit 4~15: `i_is_ext_zone` を 1 に固定（符号ビットで埋める）
- **教育的意図**: 命令のどの位置が即値として使われ、どこからが「拡張」なのかをブロック内部の配線構造として示します。

#### (3) imm_gen.v (Level 0)
全てのフォーマット拡張器を束ねる最上位です。
- **入力**: 16ビット命令コード（`i_instr`）、即値種別選択信号（`i_imm_type`）
- **出力**: 16ビット符号拡張済み即値
- **役割**: 各 `imm_gen_sext` ユニットに命令の該当ビットを配線し、最終的にMUXで1つの即値を選択します。

---

### 3. この構成による教育的メリット

1.  **符号拡張の物理的理解**:
    符号拡張を `{{12{instr[7]}}, instr[7:4]}` のようなコード上の記述ではなく、物理的な「ビットの複製と選択」として回路図から学べます。
2.  **ISAフォーマットの視覚化**:
    「I型はここ、B型はここからビットを取る」というISAの定義が、独立した4つのブロック（sext4_i, sext12_b等）として論理合成後の回路図に現れます。
3.  **パターンの再利用**:
    「ビット素子を並べてワードを作る」というCPU設計の共通パターン（Register FileやPC Regと同様）がここでも繰り返されるため、アーキテクチャの一貫性を理解しやすくなります。
