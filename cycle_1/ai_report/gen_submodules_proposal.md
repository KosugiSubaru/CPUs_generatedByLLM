教育用シングルサイクルCPUとして、ISAの命令セットを効率的に処理し、かつ内部構造が視覚的に理解しやすい7つのサブモジュール構成を提案します。

---

### 1. 提案するサブモジュール構成

| モジュール名 | 役割 |
| :--- | :--- |
| **1. Program Counter (PC)** | 次に実行する命令のメモリアドレス（`o_addr_to_imem`）を保持するレジスタです。 |
| **2. Instruction Decoder (ID)** | `i_instr` を解析し、レジスタ番号、即値、および各部を動かす制御信号（ALUの演算種別、メモリ書き込み有効化など）を生成します。 |
| **3. Register File (RF)** | 16個の16bitレジスタを管理します。2つのレジスタを同時に読み出し、1つのレジスタに書き込みます。※0番レジスタは常に0を出力。 |
| **4. Immediate Generator (ImmGen)** | 命令形式（I-type, Branch, Jump等）に応じて、命令内の特定のビット列を抽出し、16bitに**符号拡張**して即値を生成します。 |
| **5. ALU (Arithmetic Logic Unit)** | 加減算、論理演算、シフト演算を行います。演算結果とともに、フラグ（Z, N, V）を計算します。 |
| **6. Flag Register** | ALUが生成したフラグを1クロック保持します。分岐命令はこの「1つ前の命令の結果」を参照して条件判断を行います。 |
| **7. Branch Control / PC Logic** | 次のPCの値を決定します。通常は `PC + 2` ですが、分岐（Branch）成立時やジャンプ（JAL/JALR）時は計算されたアドレスを選択します。 |

---

### 2. CPUの動作プロセスと相互作用

シングルサイクルCPUとして、1クロックの中で以下の「情報の流れ」が発生します。

#### ① Fetch (命令取り出し)
*   **PC** が現在の命令アドレスを出力し、命令メモリ（`imem`）から命令 `i_instr` が読み込まれます。
*   ※提供された `imem` は同期読み出しのため、厳密には「前のサイクルの終わりに確定したPC」に対応する命令が現在届いている状態となります。

#### ② Decode & Read (解読と読み出し)
*   **Instruction Decoder** が命令のビット列（opcode）を見て、「これはADD命令だ」「これはLOAD命令だ」と判断します。
*   **Register File** は、命令に含まれる `rs1`, `rs2` の番号に基づき、保持しているデータを ALU へ送り出します。
*   **ImmGen** は、命令内の即値フィールドを抽出し、符号拡張して準備します。

#### ③ Execute (実行)
*   **ALU** が、デコーダーの指示に従って演算を行います。
    *   R-typeなら：`rs1` と `rs2` を演算。
    *   I-type（addi等）なら：`rs1` と **ImmGen** の即値を演算。
*   演算と同時に、新しいフラグ（Z, N, V）が計算され、**Flag Register** に送られます。

#### ④ Memory / Write Back (メモリ・書き戻し)
*   **データメモリ(dmem)へのアクセス**:
    *   `load` 命令：ALUが計算したアドレスからデータを読み出します。
    *   `store` 命令：ALUが計算したアドレスへ、`rs2` のデータを書き込みます（`o_dmem_wen`を有効化）。
*   **レジスタへの書き戻し**:
    *   演算結果（ALU出力）またはメモリから読み出したデータ（`i_data_from_dmem`）を、対象のレジスタ `rd` に保存します。

#### ⑤ Next PC Calculation (次アドレスの決定)
*   **Branch Control** が、**Flag Register** に保存されている「1クロック前のフラグ」を確認します。
*   条件（`bz`ならZ=1など）が成立していれば `PC + imm`、成立していなければ `PC + 2` を次のPC値として設定します。
*   `jal` や `jalr` の場合は、即値やレジスタ値を元にジャンプ先を計算します。

### 教育的なポイント：視覚化のヒント
この構成にすることで、以下のような「データの通り道（データパス）」を視覚化できます。
*   **算術演算**: RegFile → ALU → RegFile というループ。
*   **ロード/ストア**: RegFile(Addr) → ALU → Dmem → RegFile(Data) というメモリを介した流れ。
*   **フラグの遅延**: フラグが一度レジスタに格納され、次の命令の条件分岐に使われる様子。

各モジュール間の接続（ワイヤ）を `top` モジュールのデバッグポート（`o_debug_*`）に接続することで、波形シミュレータ上で「今どのレジスタが使われ、どんな演算がされているか」を一目で追えるようになります。
