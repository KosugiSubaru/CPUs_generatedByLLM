「instruction_decoder」モジュールを、命令コードの解析過程が視覚的に理解できるよう、パターン構造化（16本の命令有効信号への展開）を用いて階層化します。

論理合成後の回路図で「どの命令が現在アクティブか」が16本のラインとして見えるように構成するのが教育的なポイントです。

---

### 1. 階層構造とファイル一覧

| 階層レベル | ファイル名 | 役割 |
| :--- | :--- | :--- |
| **Level 0 (基本素子)** | `instruction_decoder_op_match.v` | 入力された4bitのopcodeが、指定された定数と一致するかを判定する比較器。 |
| **Level 1 (構造化ユニット)** | `instruction_decoder_op_bus.v` | `generate`文を用いて `op_match` を16個並列化。Opcodeを16本の命令個別信号（is_add, is_sub等）に変換する。 |
| | `instruction_decoder_field_extractor.v` | 命令ビット列から `rd`, `rs1`, `rs2`, `imm` などの各フィールドを単純に切り出す。 |
| | `instruction_decoder_control_logic.v` | 16本の命令信号を元に、レジスタ書き込み有効、メモリ書き込み有効などの各制御信号を論理和等で生成する。 |
| **Level 2 (サブモジュール・トップ)** | `instruction_decoder.v` | 上記を統合し、命令(16bit)から全ての制御信号とレジスタアドレスを出力する。 |

---

### 2. 各モジュールの詳細説明

#### Level 0: 基本素子
*   **instruction\_decoder\_op\_match.v**
    *   **役割**: 4ビットの比較器。
    *   **回路図での見え方**: 単一の比較ブロック。これが16個並ぶことで「デコード」の根拠が明確になります。

#### Level 1: パターン構造化と論理生成
*   **instruction\_decoder\_op\_bus.v**
    *   **役割**: `generate`文を使用し、0000から1111までの各オペコードに対応する `op_match` を一斉にインスタンス化します。
    *   **回路図での見え方**: 16個のブロックが縦に並び、そこから16本の信号線（命令バス）が右へ伸びる壮観な構造になります。これにより「どの命令が選択されているか」が1ビットのフラグ群として視覚化されます。
*   **instruction\_decoder\_field\_extractor.v**
    *   **役割**: ISAの定義に基づき、ビットをスライスします（例: `rd = instr[15:12]`）。
    *   **回路図での見え方**: 結線の束（バス・スプリッタ）として表現されます。
*   **instruction\_decoder\_control\_logic.v**
    *   **役割**: 命令バスを受け取り、各部の「有効/無効」を決定します。
        *   例: `reg_write_en` = `is_add | is_sub | is_and | ... | is_load | is_jal`
    *   **回路図での見え方**: 命令信号を束ねる大きなORゲート群として視覚化されます。

#### Level 2: サブモジュール・トップ
*   **instruction\_decoder.v**
    *   **役割**: 命令入力を受け取り、以下の信号をCPU内部に分配します。
        1.  **レジスタアドレス**: `rd`, `rs1`, `rs2` 番号。
        2.  **ALU制御信号**: 演算種別選択。
        3.  **メモリ制御信号**: `mem_wen`, `mem_to_reg` など。
        4.  **PC制御信号**: `pc_sel` (通常/分岐/ジャンプ)。
        5.  **即値情報**: そのままImmGenへ渡すか、ここで抽出した値を渡します。

---

### 3. 教育的なポイント：視覚化のメリット

1.  **「デコード」の可視化**: 一般的な `case` 文による記述は論理合成すると複雑なゲートの塊になりがちですが、このように `op_match` を並列に並べる手法をとると、**「命令の種類数だけ判定器がある」**というハードウェアの基本原則が回路図上で非常に分かりやすくなります。
2.  **制御ロジックの透明化**: 「なぜこの命令でレジスタが書き換わるのか？」が、ORゲートの入力元を辿ることで「is_add信号が立っているからだ」と直感的に理解できるようになります。
3.  **拡張性**: 新しい命令をISAに追加する際、`op_bus` の `generate` 範囲を調整し、`control_logic` にOR入力を一つ増やすだけという、規則的な作業で対応できることが学べます。
