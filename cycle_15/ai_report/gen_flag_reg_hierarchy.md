サブモジュール `flag_reg` について、CPUの状態（フラグ）を保存し、条件分岐命令に引き渡すための階層設計を提案します。

このモジュールは、ALUで計算された「最新のフラグ」を、**「次の命令が参照するためのフラグ」**としてクロック同期で保持する役割を持ちます。教育的視点から、各フラグ（Z, N, V）が個別のレジスタユニットとして独立して動く様子を視覚化します。

---

### 1. 階層構造一覧

| 階層レベル | ファイル名 | 役割 |
| :--- | :--- | :--- |
| **Level 0** | `flag_reg.v` | 最上位。Z, N, Vの各フラグユニットを統合する。 |
| **Level 1** | `flag_reg_unit.v` | 1ビットのフラグ保持ユニット。書き込み有効信号（Enable）により更新を制御する。 |
| **Level 2** | `flag_reg_dff.v` | クロック同期の1ビットD型フリップフロップ。 |
| **Level 2** | `flag_reg_mux2.v` | 1ビットの2入力セレクタ。Enable信号に応じて「保持」か「更新」かを選択する。 |

---

### 2. 各モジュールの役割と階層の詳細説明

#### Level 0: 統合
- **`flag_reg.v`**
  - **役割:** ALUから出力された最新のフラグ信号（Z, N, V）と、Control Unitからの更新許可信号（`i_flag_wen`）を入力し、保存されたフラグ値をCPU全体（特にNext PC Logic）へ出力します。
  - **パタン構造:** `generate` 文を使用して、3つのフラグ（Z, N, V）を `flag_reg_unit` として並列にインスタンス化します。
  - **視覚的特徴:** 合成図において、3本の信号線がそれぞれ独立した箱（レジスタ）に吸い込まれ、保持される様子が明確に分かります。

#### Level 1: 制御付保持ユニット (Unit)
- **`flag_reg_unit.v`**
  - **役割:** 特定の命令（addiやsubなど、フラグを更新する命令）の時だけALUの結果を取り込み、それ以外の命令の時は現在の値を維持（Hold）するロジックを持ちます。
  - **論理構成:** `flag_reg_mux2` と `flag_reg_dff` を組み合わせて「フィードバック付レジスタ」を構成します。
  - **教育的意図:** 「フラグが更新される命令」と「更新されない命令（例：loadやjump）」の違いを、ハードウェアがどのように実現しているかを視覚的に示します。

#### Level 2: 基礎素子 (Primitives)
- **`flag_reg_dff.v`**
  - **役割:** クロック立ち上がりで入力を保存する、回路の最小記憶単位です。
- **`flag_reg_mux2.v`**
  - **役割:** `Enable` 信号が `1` なら「新しいフラグ値」を、`0` なら「自分自身の現在の出力値」をDFFの入力へ流します。

---

### 3. パタン構造化（generate文）の適用イメージ

`flag_reg.v` 内で、ビットごとに構造を繰り返す例です。

```verilog
// flag_reg.v 内のイメージ
wire [2:0] w_flags_in  = {i_z_in, i_n_in, i_v_in};
wire [2:0] w_flags_out;

genvar i;
generate
    for (i = 0; i < 3; i = i + 1) begin : gen_flag_bits
        flag_reg_unit u_flag_unit (
            .i_clk   (i_clk),
            .i_rst_n (i_rst_n),
            .i_en    (i_flag_wen), // 更新有効信号
            .i_d     (w_flags_in[i]),
            .o_q     (w_flags_out[i])
        );
    end
endgenerate
```

### この構成のメリット
1.  **「1クロック遅れて分岐する」仕様の視覚化:** 
    ALUの結果が一度 `flag_reg` に入り、そこから分岐ロジックへ伸びる配線を見ることで、ISAノートにある「1クロック前の結果を参照する」という意味が物理的に理解できます。
2.  **選択的更新の理解:** 
    すべての命令でフラグが変わるわけではない（Enable制御がある）ことを、ユニット内のセレクタ（MUX2）の存在によって示せます。
3.  **モジュール性の維持:** 
    将来的にキャリーフラグ(C)などを追加したい場合も、`generate` のループ回数を変更するだけで視覚的・構造的な美しさを保ったまま拡張可能です。
