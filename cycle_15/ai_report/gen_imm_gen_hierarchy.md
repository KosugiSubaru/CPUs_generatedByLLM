サブモジュール `imm_gen` について、ISAで定義された複数の即値形式（4bit, 8bit, 12bit）とそれらの符号拡張処理を、回路図上で直感的に分離・視覚化するための階層設計を提案します。

この設計では、各命令フォーマットに応じた「抽出・拡張ユニット」を独立させ、それらをセレクタで切り替える構造を採ります。

---

### 1. 階層構造一覧

| 階層レベル | ファイル名 | 役割 |
| :--- | :--- | :--- |
| **Level 0** | `imm_gen.v` | 最上位。各抽出ユニットと最終セレクタを統合する。 |
| **Level 1** | `imm_gen_extractor_4bit_low.v` | `instr[7:4]` を抽出し、16bitへ符号拡張する（addi, load, jalr用）。 |
| **Level 1** | `imm_gen_extractor_4bit_high.v` | `instr[15:12]` を抽出し、16bitへ符号拡張する（store用）。 |
| **Level 1** | `imm_gen_extractor_8bit.v` | `instr[11:4]` を抽出し、16bitへ符号拡張する（loadi, jal用）。 |
| **Level 1** | `imm_gen_extractor_12bit.v` | `instr[15:4]` を抽出し、16bitへ符号拡張する（blt, bz用）。 |
| **Level 1** | `imm_gen_mux_4to1_16bit.v` | 4つの拡張済み即値から、制御信号に基づき1つを選択する。 |
| **Level 2** | `imm_gen_mux_2to1_16bit.v` | 16ビットの2入力セレクタ。4to1 MUXを構成する基本要素。 |

---

### 2. 各モジュールの役割と階層の詳細説明

#### Level 0: 統合
- **`imm_gen.v`**
  - **役割:** CPUコアから命令全体（`i_instr`）と、どの即値形式を使用するかを決定する制御信号（`i_imm_sel`）を受け取り、最終的な16ビット即値を出力します。
  - **視覚的特徴:** 合成後の回路図において、4つの独立した抽出器が並列に配置され、その出力が1つの大きなセレクタに収束する様子が確認できます。

#### Level 1: 抽出と符号拡張 (Extractor)
ISAの定義に基づき、ビット位置の異なる即値を整理します。
- **`imm_gen_extractor_4bit_low.v`** / **`high.v`**
  - **役割:** 4ビットのフィールドを16ビット幅に広げます。最上位ビット（符号ビット）を上位12ビットすべてにコピー（符号拡張）します。
  - **教育的意図:** 命令のどこからビットを連れてきているかを、モジュール名と配線で明示します。
- **`imm_gen_extractor_8bit.v`** / **`12bit.v`**
  - **役割:** それぞれ8ビット、12ビットのフィールドを同様に16ビットへ拡張します。

#### Level 1 & 2: 選択ロジック (Multiplexer)
- **`imm_gen_mux_4to1_16bit.v`**
  - **構造化:** **`generate` 文**を使用して、`imm_gen_mux_2to1_16bit` を3つ（または4つ）組み合わせてツリー構造を構築します。
  - **教育的意図:** 複雑な4択処理を単純な2択の連鎖として視覚化し、論理合成時のゲート爆発を防ぎます。
- **`imm_gen_mux_2to1_16bit.v`**
  - **役割:** セレクト信号に基づき、16ビット幅のデータを切り替えます。

---

### 3. パタン構造化（generate文）の適用イメージ

`imm_gen_mux_4to1_16bit.v` において、2入力セレクタを階層的に配置する例です。

```verilog
// imm_gen_mux_4to1_16bit.v 内の構造化イメージ
wire [15:0] w_mux_stage_mid[1:0];
genvar i;
generate
    // ステージ1: 4入力を2つの中間出力へ絞り込む
    for (i = 0; i < 2; i = i + 1) begin : gen_mux_s1
        imm_gen_mux_2to1_16bit u_mux_s1 (
            .i_sel  (i_sel[0]),
            .i_d0   ( (i == 0) ? i_d0 : i_d2 ),
            .i_d1   ( (i == 0) ? i_d1 : i_d3 ),
            .o_data (w_mux_stage_mid[i])
        );
    end
    
    // ステージ2: 最終的な1出力を決定
    imm_gen_mux_2to1_16bit u_mux_s2 (
        .i_sel  (i_sel[1]),
        .i_d0   (w_mux_stage_mid[0]),
        .i_d1   (w_mux_stage_mid[1]),
        .o_data (o_data)
    );
endgenerate
```

### この構成のメリット
1.  **ビットフィールドの対応が明確:** `store` 命令だけが `[15:12]` という特殊な位置の即値を使うことなどが、専用の `extractor_4bit_high` ブロックの存在によって一目で理解できます。
2.  **符号拡張の視覚化:** 狭いビット幅（4, 8, 12）が共通の広い幅（16）に変換されるプロセスが、独立したブロックとして表現されるため、データパスの概念が掴みやすくなります。
3.  **制御信号との整合性:** `Control Unit` からの `imm_sel` 信号が、どのように即値の「解釈」を切り替えているかが、MUXへの配線を通じて論理的に示されます。
